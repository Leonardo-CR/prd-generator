# OpenSpec Bridge

Reads the output of the `module-breakdown` skill and produces OpenSpec-compatible
`spec.md` files for each module, plus the project `openspec/config.yaml`.

This is **Stage 3** of the development pipeline:

```
[PRD] → [Module Breakdown] → [OpenSpec Bridge] → [OpenSpec Workflow] → [Code]
```

Once this skill completes, the user can use OpenSpec's native commands
(`/openspec:proposal`, etc.) directly on each module spec.

---

## Pre-condition Check

Before starting, verify both of the following exist:

1. `prd-generator-utils/PRD-*.md` — the project PRD
2. `prd-generator-utils/modules/MODULE-INDEX.md` — the module index

**If PRD is missing:**
> "No encontré el PRD en `prd-generator-utils/`. Primero necesitas generar el PRD
> con el skill `prd-generator`. ¿Quieres hacerlo ahora?"

**If MODULE-INDEX.md is missing:**
> "No encontré el Module Index en `prd-generator-utils/modules/`. Primero necesitas
> correr el skill `module-breakdown` para dividir el proyecto en módulos.
> ¿Quieres hacerlo ahora?"

If both exist, read them in full before proceeding. Extract:
- From PRD: project name, tech stack, domain entities, non-functional requirements
- From MODULE-INDEX: module list, types, dependency order, dependency graph

## Phase 2: Generate openspec/config.yaml

Before generating specs, create the OpenSpec project configuration. This injects
project context into every artifact OpenSpec generates.

Read the PRD's **Technical Constraints** section to extract the stack.
Read the PRD's **Non-Functional Requirements** for global rules.

Save to `openspec/config.yaml` at the project root:

```yaml
schema: spec-driven

context: |
  Project: {PROJECT_NAME}
  {One paragraph summarizing the project's purpose from the PRD Problem Statement.}

  Tech stack: {Language, framework, database, cloud — from PRD Technical Constraints}
  API style: {REST / GraphQL / gRPC — from PRD or module files}
  Platform: {Web / Mobile / Desktop / CLI}

  Modules:
  {List each module and one-line description, e.g.:}
    - database: {description}
    - auth: {description}
    - backend-api: {description}
    - frontend: {description}

  Module dependency order (build in this sequence):
  {Ordered list from MODULE-INDEX recommended build order}

  Domain entities: {Comma-separated list of entities from PRD Domain Model}

rules:
  proposal:
    - Reference the affected module spec by name
    - Include rollback plan for database or auth changes
    - Identify which other modules are impacted by this change
  design:
    - Stay consistent with the tech stack defined in context
    - Do not introduce new dependencies without noting them in the proposal
  spec:
    - Use SHALL for non-negotiable behavior, SHOULD for recommended behavior
    - Every requirement must have at least one Scenario (Given/When/Then)
    - Cross-module contracts must be explicit — name the module they depend on
```

## Phase 3: Generate spec.md Files

Process modules **one at a time**, in the dependency order from MODULE-INDEX.
For each module:

1. Read `prd-generator-utils/modules/{module-name}.md` in full
2. Transform it using the mapping rules below
3. Save to `openspec/specs/{module-name}/spec.md`
4. Confirm to the user before moving to the next module:
   > "`{module-name}/spec.md` generado. Pasando a `{next-module}`..."

Create the `openspec/specs/` folder structure if it does not exist.

---

### Transformation Mapping

| Module section | → | OpenSpec section |
|----------------|---|-----------------|
| Purpose | → | `## Purpose` (verbatim, condensed to 2-3 sentences) |
| Scope > In scope | → | First paragraph under `## Purpose` |
| Functional Specification (SPEC-XX) | → | `### Requirement:` entries |
| Acceptance Criteria items | → | `#### Scenario:` blocks under the relevant Requirement |
| Non-Functional Requirements | → | `### Requirement:` entries using SHOULD/MUST language |
| Interface Contract > API Endpoints | → | `### Requirement:` entries describing observable API behavior |
| Dependencies > Depends on | → | Cross-module references inside relevant Requirements |
| Open Questions | → | `## Open Questions` section at the bottom |

### Conversion rules

**Functional Spec → Requirement:**
- Each `SPEC-XX` becomes one `### Requirement: {name}`
- The description becomes a SHALL/MUST statement describing observable behavior
- Do NOT mention implementation details (class names, function names, libraries)
- Do NOT copy internal logic — describe what the system does, not how

**Acceptance Criteria → Scenario:**
- Each checkbox item becomes a `#### Scenario:` with Given/When/Then steps
- Binary criteria map cleanly: the "given" is the precondition, "when" is the
  trigger, "then" is the observable outcome
- Add AND lines for multiple outcomes

**Non-Functional Requirements → Requirements:**
- Performance, Security, Scalability rows become `### Requirement:` entries
- Use MUST for hard constraints (security, correctness)
- Use SHOULD for quality targets (performance thresholds, usability)
- Add at least one Scenario per NFR requirement

**Cross-module dependencies → Requirement references:**
- When a requirement depends on another module, add a note:
  `> Cross-module: depends on \`{module-name}\` spec.`
- This makes inter-module contracts explicit and traceable

## spec.md Template

Use this template for every generated spec file:

```markdown
# {Module Name} Specification

> **Module type:** `{type}`
> **Pipeline stage:** OpenSpec Bridge
> **Source:** `prd-generator-utils/modules/{module-name}.md`
> **Depends on:** {comma-separated module names, or "none"}
> **Required by:** {comma-separated module names, or "none"}

---

## Purpose

{2-3 sentences describing what this module does and its single responsibility.
Extracted from the module file's Purpose and Scope sections.}

---

## Requirements

### Requirement: {Requirement name — from SPEC-01 title}

{One sentence using SHALL/MUST describing the observable system behavior.
No implementation details.}

#### Scenario: {Happy path name}
- GIVEN {precondition — system state or user state}
- WHEN {action or event that triggers the behavior}
- THEN {observable outcome}
- AND {additional outcome if needed}

#### Scenario: {Error or edge case name}
- GIVEN {precondition}
- WHEN {action}
- THEN {expected error behavior or fallback}

> Cross-module: depends on `{other-module}` spec. *(only if applicable)*

---

### Requirement: {Next requirement — from SPEC-02 title}

{SHALL statement}

#### Scenario: {Scenario name}
- GIVEN {precondition}
- WHEN {action}
- THEN {outcome}

---

*(Repeat for each SPEC-XX from the module file)*

---

## Non-Functional Requirements

### Requirement: {NFR name — e.g., "Response time"}

The system {MUST/SHOULD} {NFR statement in observable terms}.

#### Scenario: {Verification scenario}
- GIVEN {load or context condition}
- WHEN {operation is performed}
- THEN {measurable outcome, e.g., "response is received in under 500ms"}

---

*(Repeat for each NFR row)*

---

## Open Questions

> These questions were unresolved during module breakdown. Resolve them before
> creating OpenSpec change proposals for this module.

| # | Question | Impact | Owner |
|---|----------|--------|-------|
| 1 | {Question} | {What it blocks} | {Who decides} |

*(Write "No open questions." if none exist)*
```

## Output Rules

- Save `openspec/config.yaml` at the project root.
- Save each spec at `openspec/specs/{module-name}/spec.md`.
  Use the same kebab-case names from the module files.
- Create directories as needed — never fail because a folder doesn't exist.
- Never truncate sections. Write `> No definido aún.` for empty sections.
- Never include implementation details (class names, SQL internals, library APIs)
  in spec.md — those belong in `design.md`, which OpenSpec generates later.
- Never invent requirements not present in the module files.
- Keep Scenarios short and binary — a scenario either passes or fails, no ambiguity.

## Pipeline Context

```
[PRD] → [Module Breakdown] → [OpenSpec Bridge] → [OpenSpec Workflow] → [Code]
```

The `spec.md` files produced here are **living documents**. As the project evolves,
OpenSpec's `/openspec:proposal` command will generate spec deltas that modify these
files. The initial content from this skill is version 1.0 — the baseline against
which all future changes are measured.

Key constraints to maintain across all specs:
- Entity names must match the PRD's Domain Model exactly
- Module names must match MODULE-INDEX.md exactly (used for cross-references)
- Open questions must be resolved before writing proposals that depend on them
- Dependency order from MODULE-INDEX must be respected when creating proposals
