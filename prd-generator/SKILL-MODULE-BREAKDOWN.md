# Module Breakdown

Reads a PRD generated by the `prd-generator` skill and decomposes the project
into discrete development modules. Each module gets its own `.md` file with
sufficient context to be developed independently or in coordination with related
modules.

This is **Stage 2** of the development pipeline:

```
[PRD] → [Module Breakdown] → [User Stories] → [OpenSpec] → [Code]
```

---

## Pre-condition Check

Before starting, verify that a PRD file exists at `prd-generator-utils/PRD-*.md`.

- **If found:** Read it in full. Extract: project name, domain model (entities),
  functional requirements (FR-XX), technical constraints, and non-goals.
- **If not found:** Tell the user:
  > "No encontré un PRD en `prd-generator-utils/`. El Module Breakdown necesita
  > un PRD como base. ¿Quieres que generemos el PRD primero?"
  Then stop.

---

## Phase 1: Module Proposal

Based on the PRD content, propose a list of modules. Use the functional requirements,
domain model, and technical stack to infer natural boundaries.

### Module type catalogue

Use these types as reference — not all projects need all types:

| Type | When to include | Typical sub-concerns |
|------|-----------------|----------------------|
| `database` | Any persistence layer | Entities, relations, indexes, migrations |
| `auth` | Authentication or roles exist | Strategy, roles, permissions, token lifecycle |
| `backend-api` | Server-side logic / REST / GraphQL | Endpoints, business logic, validations |
| `frontend` | UI layer (web or mobile) | Views, user flows, state, components |
| `integrations` | External APIs or services | Contracts, error handling, rate limits |
| `infrastructure` | Deployment, CI/CD, environments | Hosting, env vars, pipelines |
| `admin-panel` | Back-office or internal tools | CRUD interfaces, permissions |
| `notifications` | Email, SMS, push, webhooks | Triggers, templates, delivery guarantees |
| `background-jobs` | Async processing, queues, crons | Job types, retry logic, scheduling |

### Proposal format

Present the proposed modules to the user in this format:

> Basándome en el PRD, propongo dividir el proyecto en los siguientes módulos:
>
> | # | Módulo | Tipo | Descripción breve |
> |---|--------|------|-------------------|
> | 1 | `{module-name}` | `{type}` | {One sentence} |
> | 2 | `{module-name}` | `{type}` | {One sentence} |
> ...
>
> ¿Quieres agregar, quitar o renombrar algún módulo antes de continuar?

**Wait for user confirmation or adjustments before proceeding.**
Apply any changes the user requests. Once approved, lock the module list and
move to Phase 2.

---

## Phase 2: Module Interviews

Interview the user **one module at a time**. For each module, present ONE group
of questions, wait for answers, then present the next group.
NEVER ask questions for two different modules in the same message.

Process modules in dependency order: foundational modules first
(e.g., `database` before `backend-api`, `auth` before `frontend`).

For each module, run the applicable question groups below based on its type.
Skip groups that don't apply to the module type.

---

### Question Groups by Module Type

#### Group DB-A — Entities & Attributes (for `database` modules)

1. **¿Cuáles son las entidades principales de la base de datos?**
   *(List all tables/collections; name them in singular)*

2. **Por cada entidad, ¿cuáles son sus atributos clave?**
   *(Types, required/optional, unique constraints)*

3. **¿Qué campos deben tener todas las entidades por defecto?**
   *(e.g., id, created_at, updated_at, soft delete flag)*

#### Group DB-B — Relations & Rules (for `database` modules)

4. **¿Cuáles son las relaciones entre entidades? (1:1, 1:N, N:M)**

5. **¿Hay reglas de integridad importantes?**
   *(Cascades, restrictions on delete, required foreign keys)*

6. **¿Usarás migraciones? ¿Hay datos semilla (seeds) necesarios?**

---

#### Group AUTH-A — Strategy & Roles (for `auth` modules)

1. **¿Qué estrategia de autenticación usarás?**
   *(JWT, sessions, OAuth/SSO, magic links, API keys)*

2. **¿Habrá roles de usuario? ¿Cuáles y qué puede hacer cada uno?**

3. **¿Qué rutas o recursos son públicos vs. protegidos?**

#### Group AUTH-B — Token & Session Lifecycle (for `auth` modules)

4. **¿Cuánto tiempo dura una sesión o token? ¿Habrá refresh tokens?**

5. **¿Cómo se maneja el cierre de sesión y la revocación de acceso?**

6. **¿Hay requerimientos de seguridad específicos?**
   *(2FA, rate limiting en login, bloqueo por intentos fallidos)*

---

#### Group API-A — Endpoints & Resources (for `backend-api` modules)

1. **¿Cuáles son los recursos principales que expondrá la API?**
   *(List them; map to domain entities where possible)*

2. **Por cada recurso, ¿qué operaciones necesita?**
   *(CRUD estándar, operaciones custom como `/activate`, `/send`, etc.)*

3. **¿La API es REST, GraphQL, o mixta? ¿Habrá versionado?**

#### Group API-B — Logic & Validations (for `backend-api` modules)

4. **¿Qué lógica de negocio relevante vive en cada endpoint?**
   *(Cálculos, transformaciones, reglas de validación)*

5. **¿Qué errores deben manejarse explícitamente?**
   *(Not found, forbidden, conflict, external service failure)*

6. **¿Hay endpoints que disparen efectos secundarios?**
   *(Emails, jobs, notificaciones, actualizaciones en cascada)*

---

#### Group FE-A — Views & Flows (for `frontend` modules)

1. **¿Cuáles son las vistas o pantallas principales?**
   *(List them with a one-line description of purpose)*

2. **¿Cuál es el flujo principal del usuario de inicio a fin?**
   *(Step-by-step happy path)*

3. **¿Hay flujos alternativos importantes?**
   *(Error states, empty states, onboarding, edge cases)*

#### Group FE-B — State & Components (for `frontend` modules)

4. **¿Qué estado global necesita la app?**
   *(Auth session, cart, filters, notifications, etc.)*

5. **¿Hay componentes reutilizables que ya identificas?**
   *(Tables, forms, modals, cards, etc.)*

6. **¿Cuáles son los requerimientos de UX más críticos?**
   *(Loading states, optimistic updates, responsiveness, accesibilidad)*

---

#### Group INT-A — Contract & Error Handling (for `integrations` modules)

1. **¿Qué servicios externos se integran y para qué?**
   *(Name each service and its purpose)*

2. **¿Cómo se autentican esas integraciones?**
   *(API keys, OAuth, webhooks inbound)*

3. **¿Qué pasa si el servicio externo falla o tarda demasiado?**
   *(Retry, fallback, error visible al usuario, job reintentable)*

#### Group INT-B — Data Mapping (for `integrations` modules)

4. **¿Qué datos entran y salen de cada integración?**
   *(Request/response shape, transformaciones necesarias)*

5. **¿Hay rate limits o cuotas que debas respetar?**

---

#### Group INFRA-A — Hosting & Environments (for `infrastructure` modules)

1. **¿Dónde se desplegará la aplicación?**
   *(Cloud provider, VPS, serverless, containers)*

2. **¿Cuántos entornos habrá?**
   *(local, staging, production — y sus diferencias clave)*

3. **¿Qué variables de entorno son necesarias?**
   *(List them; mark which are secrets)*

#### Group INFRA-B — CI/CD & Ops (for `infrastructure` modules)

4. **¿Habrá pipeline de CI/CD? ¿Qué debe incluir?**
   *(Tests, linting, build, deploy automático)*

5. **¿Hay requerimientos de monitoreo, logs o alertas?**

---

#### Group GEN-A — General (for any module type not listed above)

1. **¿Qué responsabilidades concretas tiene este módulo?**

2. **¿Qué datos maneja o produce?**

3. **¿Qué otros módulos dependen de este, o de cuáles depende?**

#### Group GEN-B — Edge Cases & Constraints

4. **¿Cuáles son los casos borde más importantes de este módulo?**

5. **¿Hay restricciones técnicas específicas para este módulo?**

---

### Interview rules

- Ask ONE group per message. Wait for answers before the next group.
- If the user's answers are thin, ask one targeted follow-up before moving on.
- If a question doesn't apply to the specific module, skip it silently.
- If the user already answered a question earlier (from the PRD or conversation),
  extract the answer and skip re-asking.
- After finishing all groups for a module, confirm:
  > "Listo con `{module-name}`. Pasamos al siguiente módulo: `{next-module}`."

## Phase 3: Module File Generation

After completing the interview for each module, generate its `.md` file immediately
(do not wait until all modules are done). Save it to:

```
prd-generator-utils/modules/{module-name}.md
```

Create the `modules/` folder if it does not exist.

Use the module template below. Fill every field. Never leave placeholder text —
write `> No definido aún.` for genuinely unknown sections.

---

## Output Rules

- Generate each module file immediately after its interview is complete.
- Save files to `prd-generator-utils/modules/{module-name}.md`.
  Use kebab-case for file names (e.g., `backend-api.md`, `auth.md`).
- Create the `modules/` folder if it does not exist.
- Generate `MODULE-INDEX.md` only after ALL module files are done.
- Never truncate sections. Write `> No definido aún.` for empty sections.
- After saving all files, confirm to the user:

> "Los módulos de **{PROJECT_NAME}** están listos en `prd-generator-utils/modules/`.
> Puedes revisar cada archivo y resolver las preguntas abiertas antes de continuar.
> El siguiente paso es definir las **User Stories** por módulo. ¿Quieres empezar?"

---

## Pipeline Context

This Module Breakdown is **Stage 2** of the following development pipeline:

```
[PRD] → [Module Breakdown] → [User Stories] → [OpenSpec] → [Code]
```

Each module file is self-contained: a downstream AI must be able to develop the
module reading only its `.md` file plus the `MODULE-INDEX.md` for dependency context.
Write with that constraint in mind:
- Never reference "what we discussed" — all context must be in the file
- Use entity names consistent with the PRD's Domain Model (Section 4)
- Trace every capability back to a PRD functional requirement (FR-XX) where possible
- Flag all open questions explicitly — never assume answers
